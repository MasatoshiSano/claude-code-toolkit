# ワークフローガイド

## 主要ワークフロー

### 1. 新機能開発（Spec-Driven Development）

```
/spec "機能説明" → 要件定義 → 詳細設計 → タスク分解 → 実装 → /commit → /create-pr
```

**フルワークフロー:**
1. `/spec "feature description"` で要件・設計・タスクを生成
2. 各段階で承認を求められる
3. `.tmp/tasks.md` に従って実装
4. `/commit` でコミット
5. `/create-pr` でPR作成

**個別実行:**
- `/requirements "新機能"` - 要件のみ生成（既存にマージ）
- `/design` - 設計のみ生成（requirements.md を参照）
- `/tasks` - タスクのみ生成（design.md を参照）

### 2. クイックコミット & PR

```
コード変更 → /commit → (ブログ提案) → /create-pr
```

1. コードを変更
2. `/commit` で自動ステージング・コミット
3. 技術ブログ候補が検出された場合、作成を提案
4. `/create-pr` でPR作成

### 3. 技術ブログ作成

**自動提案ワークフロー:**
1. `/commit` でコミット
2. 技術ブログ候補が自動検出
3. 記事作成を選択（y/N）
4. テーマごとに対話
5. `_docs/blog/` にドラフト生成

**手動実行:**
```
/blog → 変更分析 → 記事候補提示 → 対話 → ドラフト生成
```

### 4. コード品質チェック

```
/check → 問題修正 → /security-audit → /clean
```

1. `/check` で包括的分析
2. 発見された問題を修正
3. `/security-audit` でセキュリティレビュー
4. `/clean` でフォーマット・lint修正

### 5. インクリメンタル開発

既存ドキュメントに追加する場合：

```
/requirements "追加機能" → /design → /tasks → 実装
```

- 各コマンドは既存ドキュメントに自動マージ
- 手動編集後も `/design` → `/tasks` で同期可能

## ドキュメント管理ルール

### 同期の維持
- requirements.md 編集後 → `/design` → `/tasks` を実行
- design.md 編集後 → `/tasks` を実行

### バージョン追跡
- 全ドキュメントに「改訂履歴」セクションあり
- 変更時は日付と説明を追加

### タスクライフサイクル
- 新規タスク → 「🚧 進行中の機能」セクション
- 完了タスク → 「✅ 完了した機能」セクションに移動

## `/spec` vs 個別コマンドの使い分け

**`/spec` を使う場合:**
- 大規模な新機能
- 初期プロジェクトセットアップ
- 複数コンポーネントに影響する機能
- アーキテクチャ変更が必要な機能

**個別コマンドを使う場合:**
- 小規模な機能追加
- バグ修正のドキュメント化
- インクリメンタルな改善
- 素早いイテレーション
