## 概要

`/commit`実行後や手動実行時に、**前回コミットから現在までの変更を自動分析**し、技術ブログ記事にふさわしい内容を検出して、**対話形式で複数のドラフト記事を作成**するアシスタントです。

## 主な特徴

- ✅ **自動範囲検出**: 前回コミットから現在までの変更を自動分析
- ✅ **複数記事生成**: 1つのコミット範囲に複数のテーマがあれば分割
- ✅ **素人目線**: 技術を細分化して初心者にもわかりやすく
- ✅ **対話形式**: 記事化判定後、各テーマごとに対話
- ✅ **ドラフト作成**: すぐに編集できる下書きを生成

## 実行フロー

### 1. 変更範囲の自動検出

コミット履歴を分析：
```bash
git log --oneline -10           # 最近10件のコミット
git diff HEAD~1..HEAD           # 前回コミットからの差分
git show --stat HEAD            # 最新コミットの統計
```

### 2. 技術ブログ適性の自動判定

以下の基準で自動判定：

#### 必ず記事化（推奨度: 高）
- 新機能の実装（100行以上のコード追加）
- 新しいライブラリ・技術の導入
- アーキテクチャの変更
- パフォーマンス改善（数値的な改善がある）

#### 推奨記事化（推奨度: 中）
- 大規模リファクタリング（50行以上の変更）
- バグ修正（学びがある、複雑な）
- セキュリティ対策の実装
- UX改善の実装

#### 記事化候補（推奨度: 低）
- 小規模リファクタリング
- シンプルなバグ修正
- ドキュメント更新（重要な）

### 3. テーマの抽出と細分化

**重要**: 技術を細分化して素人にもわかるように

例: 「React のパフォーマンス最適化」→ 以下に細分化
1. 「useCallback を使って関数の再生成を防ぐ方法」
2. 「useMemo でコンポーネントの再計算を減らす」
3. 「React.memo で不要な再レンダリングを防ぐ」

**細分化の原則:**
- 1記事1テーマ
- 専門用語は使わず、または説明を付ける
- 具体的なコード例を含められる粒度
- 初心者が読んで実践できるレベル

### 4. 記事化判定の提示

ユーザーに記事候補を提示：

```
📝 技術ブログの作成候補を検出しました：

1. useCallbackで関数の再生成を防ぐ方法（推奨度: 高）
   - 対象読者: React初心者
   - 推定文字数: 1500-2000字
   - 学べること: パフォーマンス改善の基本

2. APIレスポンス時間を50%短縮した方法（推奨度: 高）
   - 対象読者: バックエンド開発者
   - 推定文字数: 2000-2500字
   - 学べること: データベースクエリ最適化

3. TypeScriptの型定義でバグを未然に防ぐ（推奨度: 中）
   - 対象読者: TypeScript初学者
   - 推定文字数: 1000-1500字
   - 学べること: 型安全性の重要性

これらの記事を作成しますか？
[1] すべて作成
[2] 選択して作成
[3] 作成しない
```

### 5. テーマごとの対話

各テーマについて対話を開始：

#### 記事1: useCallbackで関数の再生成を防ぐ方法

```
Agent: この記事について質問させてください。

Q1: このuseCallbackを使うことになったきっかけは何ですか？
    - パフォーマンス問題があった
    - コードレビューで指摘された
    - 学習のため
    - その他

User: パフォーマンス問題で、ボタンクリック時にラグがありました

Agent: なるほど。具体的にどのくらい改善されましたか？

User: クリックからレンダリングまで200msから50msになりました

Agent: 素晴らしい改善ですね！この記事で特に強調したいポイントは？

User: useCallbackを使う前と後の違いを、わかりやすく説明したい

Agent: 承知しました。それではドラフトを作成します...
```

### 6. ドラフト記事の生成

各テーマごとにMarkdown形式で生成：

```markdown
# useCallbackで関数の再生成を防ぐ方法

## はじめに

こんにちは！今回はReactのuseCallbackフックについて、
実際の開発で直面したパフォーマンス問題とその解決方法を共有します。

**この記事を読むとわかること:**
- useCallbackって何？（初心者向けに解説）
- なぜ関数の再生成が問題になるのか
- useCallbackの使い方（コード例付き）
- 実際にどのくらい速くなったか（数値で検証）

## 問題: ボタンクリックが遅い！

アプリを使っていて、ボタンをクリックしても反応が遅い...
ユーザーからそんな声をいただきました。

**症状:**
- ボタンクリックから画面更新まで200ms
- カクカクした動き
- ユーザー体験が悪い

## 原因: 関数が毎回作り直されていた

[詳細な説明...]

## 解決策: useCallbackを使う

[コード例と説明...]

## 結果: 75%の高速化！

- **改善前**: 200ms
- **改善後**: 50ms
- **改善率**: 75%削減

## まとめ

useCallbackを使うことで...
```

## 自動判定の詳細基準

### コード量による判定
- 新規追加100行以上 → 推奨度: 高
- 変更50-99行 → 推奨度: 中
- 変更10-49行 → 推奨度: 低（内容次第）

### 技術的重要度による判定
- 新技術導入 → 推奨度: 高
- アーキテクチャ変更 → 推奨度: 高
- パフォーマンス改善 → 推奨度: 高（数値がある場合）
- セキュリティ対策 → 推奨度: 高
- リファクタリング → 推奨度: 中
- バグ修正 → 推奨度: 中（学びがある場合）

### キーワードによる判定
コミットメッセージやコードに以下のキーワードがあれば記事化候補：
- "パフォーマンス", "最適化", "高速化"
- "新規", "導入", "実装"
- "セキュリティ", "脆弱性"
- "リファクタリング", "改善"
- "バグ", "修正", "解決"

## 使用例

### パターン1: `/commit`後の自動提案

```bash
/commit
# → コミット完了

📝 技術ブログの作成候補を検出しました：
1. Reactのカスタムフックでロジックを再利用する方法（推奨度: 高）

作成しますか？ [y/N] y

# → 対話開始
```

### パターン2: 手動実行

```bash
/blog

# → 前回コミットから分析
# → 記事候補を提示
# → 対話開始
```

### パターン3: 範囲指定（後方互換）

```bash
/blog 2b882dd..e1b74e1

# → 指定範囲を分析
# → 記事候補を提示
```
